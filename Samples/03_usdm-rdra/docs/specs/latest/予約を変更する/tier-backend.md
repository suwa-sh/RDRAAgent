# 予約を変更する - バックエンド仕様

## 変更概要

予約変更APIを新規作成する。確定済みの予約の利用日時を変更し、予約状態を「変更」に遷移する。

## API 仕様

### 予約変更API

- **メソッド**: PUT
- **パス**: /api/reservations/{reservation_id}
- **認証**: Bearer トークン（利用者）

#### リクエスト

| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| start_datetime | datetime | Yes | 変更後の利用開始日時 |
| end_datetime | datetime | Yes | 変更後の利用終了日時 |

#### レスポンス

| フィールド | 型 | 説明 |
|-----------|---|------|
| reservation_id | string | 予約ID |
| room_id | string | 会議室ID |
| start_datetime | datetime | 変更後の利用開始日時 |
| end_datetime | datetime | 変更後の利用終了日時 |
| status | string | 予約状態（「変更」） |
| estimated_price | number | 変更後の利用料金見積 |
| updated_at | datetime | 変更日時 |

## データモデル変更

既存の予約情報テーブルを使用。追加カラムなし。

## ビジネスルール

- 予約状態が「確定」の場合のみ変更可能
- 変更後の利用開始日時は現在日時より未来でなければならない
- 変更後の利用時間は会議室の運用ルール（利用可能時間帯、最低利用時間、最大利用時間）に従う
- 同一会議室の同一時間帯に他の確定済みまたは申請中の予約が存在する場合は変更不可（自身の予約は除外）
- 変更後の予約状態は「変更」となり、オーナーの再許諾が必要

## ティア完了条件（BDD）

```gherkin
Feature: 予約を変更する - バックエンド

  Scenario: 確定済みの予約を変更する
    Given 予約「R001」が「確定」状態で存在する（room_id: R001）
    And 変更先の「2026-04-11 14:00〜16:00」に予約が入っていない
    When PUT /api/reservations/R001 に start_datetime="2026-04-11T14:00:00", end_datetime="2026-04-11T16:00:00" をリクエストする
    Then ステータスコード200が返される
    And レスポンスの status が「変更」である

  Scenario: 申請状態の予約を変更しようとするとエラーを返す
    Given 予約「R002」が「申請」状態で存在する
    When PUT /api/reservations/R002 に start_datetime="2026-04-11T14:00:00", end_datetime="2026-04-11T16:00:00" をリクエストする
    Then ステータスコード400が返される
    And エラーメッセージ「確定済みの予約のみ変更できます」が返される

  Scenario: 日時が重複する場合エラーを返す
    Given 予約「R001」が「確定」状態で存在する
    And 同一会議室に「2026-04-11 14:00〜16:00」の別の確定済み予約が存在する
    When PUT /api/reservations/R001 に start_datetime="2026-04-11T14:00:00", end_datetime="2026-04-11T16:00:00" をリクエストする
    Then ステータスコード409が返される
    And エラーメッセージ「指定日時は既に予約されています」が返される
```
