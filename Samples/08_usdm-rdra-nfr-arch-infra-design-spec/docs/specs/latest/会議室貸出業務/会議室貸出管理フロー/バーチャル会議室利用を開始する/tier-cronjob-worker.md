# バーチャル会議室利用を開始する - CronJob ワーカー仕様

## 変更概要

バーチャル会議室の利用開始を自動的にトリガーするCronJobワーカーを実装する。毎分起動して確定済みのバーチャル会議室予約のうち、利用開始時刻を過ぎたものの会議室利用レコードを作成する。

## イベント処理仕様

### VirtualRoomStartJob

- **トリガー**: CronJon スケジュール（毎分: `* * * * *`）
- **入力チャネル**: なし（DB直接参照）
- **出力チャネル**: なし（DB更新のみ）
- **AsyncAPI**: 非同期チャネルなし（CronJobのため）

#### 処理フロー

1. CronJobが毎分起動
2. DBから処理対象予約を抽出:
   - 会議室種別 = バーチャル
   - 予約状態 = 確定
   - 利用開始日時 <= NOW()
   - 会議室利用レコードが未作成（冪等性チェック）
3. 対象予約がない場合は正常終了（ログに「対象なし」を記録）
4. 各対象予約に対して Backend API の内部エンドポイントを呼び出す（POST /internal/v1/virtual-usages/start）
5. API からの成功レスポンスを受けてログに「利用開始: {reservationId}, usageId: {usageId}」を記録
6. すべての処理完了後、処理件数と成功件数をログに記録してジョブを終了

#### エラーハンドリング

| エラー種別 | リトライ | DLQ | 説明 |
|-----------|---------|-----|------|
| DBアクセスエラー（SELECT失敗） | Yes（3回） | No | 次回CronJob実行時に再処理されるためDLQ不要 |
| API呼び出しエラー | Yes（3回, 1秒間隔） | No | 次回CronJob実行で再処理。エラーログに記録 |
| 冪等チェックによるスキップ | No | No | 正常処理。ログに「既存スキップ」を記録 |

## データモデル変更

### 会議室利用 (room_usages)

| カラム | 型 | 説明 | 変更種別 |
|--------|---|------|---------|
| usage_id | VARCHAR(36) | 利用ID (UUID) | 既存 |
| reservation_id | VARCHAR(36) | 予約ID | 既存 |
| room_id | VARCHAR(36) | 会議室ID | 既存 |
| user_id | VARCHAR(36) | 利用者ID | 既存 |
| usage_status | VARCHAR(20) | 利用状態（利用開始） | 既存 |
| started_at | TIMESTAMP | 利用開始日時（CronJob実行時刻） | 既存 |

## ビジネスルール

- CronJobの実行間隔は毎分とし、利用開始時刻の遅延を最大1分以内に抑える
- Backend API の冪等性確認（SELECT FOR UPDATE）により、複数CronJobが同時実行されても二重処理を防ぐ
- エラーが発生した予約は次回CronJob実行時に再処理される（自己修復型）
- CronJobはBackend APIと同一コードベースで管理し、k8s CronJobオブジェクトとして別途デプロイする
