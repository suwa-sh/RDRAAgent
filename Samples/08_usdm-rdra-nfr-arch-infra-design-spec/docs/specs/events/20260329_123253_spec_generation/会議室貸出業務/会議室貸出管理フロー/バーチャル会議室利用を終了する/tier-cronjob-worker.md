# バーチャル会議室利用を終了する - CronJob ワーカー仕様

## 変更概要

バーチャル会議室の利用終了を自動的にトリガーするCronJobワーカーを実装する。毎分起動して利用中のバーチャル会議室会議室利用のうち、利用終了時刻を過ぎたものの状態を「利用終了」に更新する。

## イベント処理仕様

### VirtualRoomEndJob

- **トリガー**: CronJob スケジュール（毎分: `* * * * *`）
- **入力チャネル**: なし（DB直接参照）
- **出力チャネル**: なし（DB更新のみ）
- **AsyncAPI**: 非同期チャネルなし（CronJobのため）

#### 処理フロー

1. CronJobが毎分起動
2. DBから処理対象会議室利用を抽出:
   - 会議室種別 = バーチャル（予約情報JOIN）
   - 会議室利用状態 = 利用中
   - 予約の利用終了日時 <= NOW()
3. 対象会議室利用がない場合は正常終了（ログに「対象なし」を記録）
4. 各対象に対して Backend API の内部エンドポイントを呼び出す（POST /internal/v1/virtual-usages/end）
5. API からの成功レスポンスを受けてログに「利用終了: {usageId}, usageMinutes: {分数}」を記録
6. すべての処理完了後、処理件数と成功件数をログに記録してジョブを終了

#### エラーハンドリング

| エラー種別 | リトライ | DLQ | 説明 |
|-----------|---------|-----|------|
| DBアクセスエラー（SELECT失敗） | Yes（3回） | No | 次回CronJob実行時に再処理されるためDLQ不要 |
| API呼び出しエラー | Yes（3回, 1秒間隔） | No | 次回CronJob実行で再処理。エラーログに記録 |
| 冪等チェックによるスキップ | No | No | 正常処理。ログに「既存スキップ」を記録 |

## データモデル変更

### 会議室利用 (room_usages)

| カラム | 型 | 説明 | 変更種別 |
|--------|---|------|---------|
| usage_status | VARCHAR(20) | 利用状態（利用中→利用終了） | 既存（更新） |
| ended_at | TIMESTAMP | 利用終了日時（CronJob実行時刻） | 既存（更新） |
| usage_minutes | INTEGER | 実利用時間（分） | 既存（更新） |

## ビジネスルール

- CronJobの実行間隔は毎分とし、利用終了時刻の遅延を最大1分以内に抑える
- Backend API の SELECT FOR UPDATE による冪等性確認で、複数CronJobが同時実行されても二重処理を防ぐ
- バーチャル会議室の利用終了はタイマートリガーのみで発生し、手動操作は不可
- VirtualRoomStartJob（利用開始）と VirtualRoomEndJob（利用終了）は同一CronJobコンテナ内で別スケジュールとして実装する

```gherkin
  Scenario: 利用終了時刻到来のバーチャル会議室利用を検出して利用終了処理を実行する
    Given 現在時刻が「2026-04-01 16:01」で、バーチャル会議室の会議室利用U-002（利用状態: 利用中、利用終了日時: 16:00）が存在する
    When CronJobが起動して処理対象会議室利用を取得する
    Then 会議室利用U-002に対して POST /internal/v1/virtual-usages/end が呼び出され、状態が「利用終了」に更新される

  Scenario: 処理対象の会議室利用が存在しない場合、正常終了して「対象なし」ログが記録される
    Given 現在時刻に対して利用終了日時が未到来の会議室利用のみが存在する
    When CronJobが起動して処理対象を取得する
    Then 処理対象が0件と判定され、「対象なし」ログが記録されてCronJobが正常終了する
```
