# オーナー精算を実行する - バックエンドワーカー仕様

## 変更概要

月末精算処理をバッチで一括実行する。オーナー単位でトランザクションを分離し、決済機関に精算依頼を送信する。

## イベント処理仕様

### 精算バッチジョブ

- **トリガー**: CronJob（月末最終日 23:00 JST）
- **入力チャネル**: settlement-request-queue
- **出力チャネル**: settlement-result-queue
- **AsyncAPI**: [asyncapi.yaml](../../../_cross-cutting/api/asyncapi.yaml) の channels.settlement-request-queue を参照

#### 処理フロー

1. 対象月の全オーナーの利用実績を集計する
2. オーナー単位で精算額を算出する（精算ルールに従う）
3. 各オーナーの精算レコードを作成する
4. 決済機関に精算依頼メッセージを送信する
5. 精算結果を記録する

#### エラーハンドリング

| エラー種別 | リトライ | DLQ | 説明 |
|-----------|---------|-----|------|
| 決済機関タイムアウト | Yes (3回) | Yes | リトライ後もエラーの場合はDLQに転送 |
| データ不整合 | No | Yes | 手動対応が必要 |
| オーナー単位エラー | No | No | 当該オーナーをスキップし他のオーナーの処理を継続 |

## データモデル変更

### settlements

| カラム | 型 | 説明 | 変更種別 |
|--------|---|------|---------|
| settlement_id | uuid | 精算ID | 追加 |
| owner_id | uuid | オーナーID | 追加 |
| settlement_month | string | 精算月 | 追加 |
| amount | decimal | 精算金額 | 追加 |
| payment_status | string | 支払状態 | 追加 |

## ビジネスルール

- 精算ルール: 月末に会議室別の利用履歴から精算額を算出する
- オーナー単位でトランザクションを分離（個別エラーが全体に波及しない）
- ジョブ冪等性: ジョブ実行IDで重複実行を検知
