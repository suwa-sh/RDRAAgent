## RDRAナレッジ
（元ファイル: RDRA_Knowledge/1_RDRA/RDRA.md）

```
# RDRA構造の説明
- 「アクター」：システムに関わる人の役割を表す
- 「アクター群」：同じような役割を持つ「アクター」をまとめ、組織の役割を示す場合もある
- 「外部システム」：システムと連携する外部のシステムを表す
- 「外部システム群」：同じような関係を持つ「外部システム」をまとめる
- 「コンテキスト」：システムを表すモデル「情報」「状態モデル」「条件」「バリエーション」をまとめる
- 「情報」：システムが扱うビジネス上の情報を表す ビジネス上管理されているID単位に存在する
- 「状態」：情報の変化を状態として扱い、ビジネス上管理する対象となる 状態を遷移させながらビジネスが進む
- 「業務」：最上位の会社の機能（マーケティング、販売や在庫管理...） BUCを構成要素に持つ
- 「BUC」：業務の価値を実現する単位で、業務フローの単位になる 同じ業務内でも関わるアクターや外部システムの違いでBUCが分かれる
- 「アクティビティ」：意味をもったまとまった一つの作業 仕事で関りのあるものを関連させる
- 「UC」：システムを使ってまとまった一つの意味をもつ作業 システムとの接点となり関りのあるものを関連させる
- 「画面」：アクターとシステムとのインターフェースになる
- 「イベント」：外部システムとのインターフェースになる
- 「タイマー」ビジネス上の時間的な区切りを表す　２５日締め　月末　

# モデル間のつながりの規則
- 「業務」は複数の「BUC」を配下にもつ
- 「BUC」は複数の「アクティビティ」を配下にもつ
- 「アクティビティ」(仕事)を行う「アクター」がつながる
- 「アクティビティ」(仕事)でシステムを使う場合は「UC」につながる
- 「UC」は情報を操作する
- 「アクター」は画面を使ってUCを操作する
- 「外部システム」は「イベント」で「UC」と連携する
- 「UC」が定時に活動する場合に「タイマー」につながる
- 「情報」は「情報」とつながることで構造化する
- 「状態モデル」は複数の「状態」を持つ
- 「UC」が「状態」を遷移させる
- 状態遷移は「状態モデル」内で「状態」を変える
- 「コンテキスト」は「情報」、「状態モデル」、「条件」、「バリエーション」を配下に複数持つ
- 「コンテキスト」内の構成要素は同一のIDで管理される
- 「条件」は「バリエーション」と「状態モデル」の組合せで出来ている
- オブジェクトは必ず対応するメタモデルが存在し、オブジェクトのつながりはメタモデル間のつながりの規則に従う

# 定義内容は各モデルのオブジェクトとして定義する
以下に例を示す
「会員」：モデルがアクター、オブジェクトが「会員」になる
「受注」：モデルが情報、オブジェクトが「受注」になる
「受注を登録する」：モデルがUC、オブジェクトが「受注を登録する」になる
```

## 初期要望分析
（元ファイル: 0_RDRAZeroOne/phase1/初期要望分析.md）

```
# 初期要望分析

## ビジネス背景と要求
- 介護業界の複雑な環境（多様な法規、多様な働き方）に対応する必要性について記述されている
- サービススタッフの柔軟な働き方とそれに対応するスケジュール調整について記述されている

## ビジネスポリシー
- 複数の小規模事業所を一元管理する組織構造について記述されている
- スタッフとサービス利用者の自由度を保つための柔軟性の方針について記述されている
- スキル別の配分と経営パラメータの柔軟性について記述されている

## ビジネスパラメータ
- 分類対象（施設分類、働き方分類、介護会員の状況分類）について記述されている
- 分類による自由度確保の戦略について記述されている

## 業務機能
- 介護会員の属性情報管理について記述されている
- 訪問介護計画の立案と実施記録の管理について記述されている
- 介護費用の計算、請求、回収に関する業務について記述されている

## RDRA構造との対比による分析

### 明確に記述されていること
- **アクター候補**：介護会員（サービス利用者）、サービススタッフ
- **外部システム候補**：法規制関連の要件の存在
- **業務機能**：会員管理、計画立案、実施記録管理、費用請求
- **情報**：介護会員属性、スケジュール情報、スキル情報、費用情報、実施記録

### 記述されていないこと
- **状態モデル**：情報がどのような状態を遷移するかについて記述されていない
- **BUC（ビジネスユースケース）**：業務がどのような単位で分かれるか、具体的な操作フローが記述されていない
- **アクティビティ**：業務がどのような活動に分解されるかが記述されていない
- **UC（ユースケース）**：システムとの具体的な相互作用内容が記述されていない
- **画面**：ユーザーインターフェースの具体的な要件が記述されていない
- **条件とバリエーション**：ビジネスの各シナリオと条件分岐が記述されていない
- **タイマー**：定時処理や時間的な区切りについて記述されていない

## 分析結論
初期要望は、ビジネスの背景、組織方針、主要な業務機能について記述されている。しかし、これらの業務を実現するための具体的なシステムの状態遷移、操作フロー、条件分岐、インターフェース定義はまだ記述されていない。次段階では、記述されている業務機能をRDRA構造のより詳細なモデル要素（状態、BUC、UC、画面、条件）に分解して定義する必要がある。
```
